Kerberos 票据攻击

一，概述

** 票据攻击（PTT）不是用来攻击的，而是一种权限维持，票据攻击大概分为四种黄金票据、白银票据、钻石票据、蓝宝石票据 **

黄金票据（Golden Ticket）- TGT 黄金票据（Golden Ticket）是指在 Windows 域环境中，利用 Kerberos 票据伪造技术获得的高度权限访问凭证。通过伪造黄金票据，攻击者可以绕过验证机制，获得对域内的资源可长期、不受限制的访问权限。

白银票据（Silver Ticket）- ST 白银票据（Silver Ticket）是指在 Windows 域环境中，利用 Kerberos 票据伪造技术获得的权限访问凭证。通过伪造白银票据，攻击者可以模拟其他用户或服务账号的身份，并访问受限资源。

钻石票据（Diamond Ticket）- DT 钻石票据（Diamond Ticket）是指在 Windows 域环境中，利用 NTLM 哈希值伪造技术获得的高权限访问凭证。通过伪造钻石票据，攻击者可以绕过验证机制，获得对域内资源的长期、不受限制的访问权限。

蓝宝石票据（Sapphire Ticket） 蓝宝石票据（Sapphire Ticket）是一种基于 Kerberos 协议的攻击技术，与钻石票据类似，但其核心区别在于 PAC（Privilege Attribute Certificate，特权属性证书）的修改方式。蓝宝石票据不是伪造的，而是通过合法请求获得的票据。

黄金票据相关试题

什么是黄金票据攻击

黄金票据攻击是利用 Kerberos 认证机制中的漏洞，通过获取****KRBTGT 账户的密码哈希，伪造 TGT（身份验证票据），从而绕过认证，获取对域内资源的访问权限。

如何防御

  1. 限制域管理员权限（如限制管理员登录到除了域控服务器的其他机器）
  2. 不使用默认的管理员administrator
  3. 禁用 krbtgt 账户，或定期修改的密码
  4. 异常监控

**krbtgt（Kerberos Ticket Granting Ticket）**是 Windows 域控制器（DC）在创建域时自动生成的内置账户，不可删除；

是 Kerberos 认证体系的 “核心账户”—— 专门用来加密 / 签名 Kerberos 的 TGT（票据授予票据），相当于域内 Kerberos 认证的 “根密钥”；

账户默认隐藏在 AD（活动目录）中，普通域用户看不到，只有域管理员能通过特定工具（如 mimikatz、AD 用户和计算机（高级视图））查到.

如果你发现系统已经遭受了黄金票据攻击，你会怎么做？

回答：首先，我会立即****快速连续更改 KRBTGT 密码两次，使所有现有的黄金票据失效。然后，我会进行全面的系统审计，检查特权账户的访问记录，查找攻击者的痕迹。同时，我会监控系统行为，防止攻击者进一步行动。

三，白银票据

概念

在请求票据发放过程中，Client 与 TGS 通信获取 ServerTicket（ST)，拥有 Ticket 就可以访问对应服务器资源。而当攻击者 拥有 ServerNTLMHash 时，就可以伪造一个 Ticket，直接与 Server 通信，让其验证。 不需要去 AS 中申请 TGT，也不需要向 TGS 中申请 ST。相当于直接伪造身份证。 白银票据也是权限维持，所以，也需要通过域管理员获取域内对应的凭证信息。

机器账户

https://www.freebuf.com/articles/network/283767.html

机器账号、计算机账号，所有加入域的主机都会有一个机器用户，用户名为机器名加$，如：WIN7$、WINXP$。在域环境中，普通域用户最多可以创建 10 个计算机账户，但是本地账号无法创建计算机账户，机器用户就是一种域用户。 默认情况下，加入域的机器默认在 CN=Computers 这个容器里面

机器用户跟 SYSTEM 用户的关系

其实****本地用户 SYSTEM 就对应于域内的机器用户（权限高），在域内的用户名就是机器名加 $，比如 WIN7，他的机器名是 WIN7，那么他在域内登录的用户名就是 WIN7$。

当我们拿下一台内网主机后，发现没有域内用户，这个时候我们将当前用户提升到 SYSTEM，就可以在域内充当机器用户了。但是提升到 SYSTEM 还需要利用一些提权方法。

白银票据相关面试题

什么是白银票据攻击？ 白银票据攻击是利用已知的服务（机器）账户密码哈希值，伪造 Kerberos 服务票据（TGS），从而绕过认证访问特定服务器上的特定服 务。

白银票据和黄金票据有什么区别？ 白银票据是伪造 TGS，只能访问特定服务（机器）；黄金票据是伪造 TGT，可以访问所有服务。白银票据需要服务（服务）账户的哈希值，而黄金票据需要 krbtgt 账户的哈希值。

如何防御白银票据攻击？ 可以通过限制服务账户权限；监控和审计服务账户行为；定期更换密码‘；

以及启用 PAC（Privilege Attribute Certificate，特权属性证书）进行****DC 签名 + 加密，来验证** **

等措施来防御白银票据攻击。

如何检测白银票据攻击？ 可以通过监控目标服务器上的事件日志（如事件 ID4624、4634、4672）来检测白银票据攻击

异常日志链（白银票据攻击）
日志序列 4624（登录类型 = 3/9，网络 / 票据登录 + 账户名 = administrator + 登录进程 = Kerberos ）→ 4672（特权登录）→ 4634（秒级注销）
核心异常点 1. 登录类型非本地；2. 源 IP 是异常机器；3. 4624 后立即触发 4672；4. 会话时长极短

黄金票据 vs 白银票据

原理对比

对比维度 黄金票据（Golden Ticket） 白银票据（Silver Ticket）
1. 伪造的票据类型 伪造****TGT（票据授予票据）——Kerberos 认证第一步的核心凭证 伪造****ST（服务票据）——Kerberos 认证第三步的核心凭证（跳过 DC 直接伪造）
2. 核心加密 / 签名依据 用****krbtgt 账户的 NTLM 哈希加密（krbtgt 是 DC 专属的 Kerberos 核心账户，全域唯一） 用****目标服务器机器账户（如 WIN2016$）的 NTLM 哈希加密（每台机器有自己的机器账户哈希）
3. 是否依赖 DC（域控） 完全不依赖！伪造 TGT 后，攻击者可直接向 DC 请求任意 ST（DC 会信任伪造的 TGT，因为用了 krbtgt 哈希） 完全绕开 DC！直接伪造 ST 发给目标服务器，全程不跟 DC 交互（目标服务器验证 ST 时，用自己的机器账户哈希解密，会误认为合法）
4. 权限覆盖范围 **域级别权限 —— 伪造 TGT 后可请求任意服务器的任意 ST，**通杀整个域内所有资源 单机 + 单服务权限 —— 只能访问****指定目标服务器的指定服务（如 Win2016 的 CIFS 共享），无法跨机器 / 跨服务
5. PAC 验证的作用 伪造的 TGT 中 PAC 由攻击者构造，需绕过 DC 的 PAC 签名验证（防御核心是强制 DC 签名 PAC） 伪造的 ST 中 PAC 由攻击者构造，需绕过目标服务器的 PAC 验证（防御核心是强制服务器向 DC 校验 PAC）
6. 核心攻击逻辑 冒充域管理员，欺骗 DC 发放任意 ST，实现全域控制 冒充任意用户，欺骗目标服务器，直接访问指定服务，实现定向控制

相关命令区别

黄金票据命令：

  • 核心参数：/krbtgt:krbtgt哈希 → 对应 “用 krbtgt 哈希加密伪造 TGT”；
  • 无/target//service → 对应 “全域通杀，无需指定目标 / 服务”。

白银票据命令：

  • 核心参数：/rc4:WIN2016$哈希 → 对应 “用目标机器账户哈希加密伪造 ST”；
  • 必须有/target//service → 对应 “只能访问指定机器的指定服务”。

四，钻石票据

概念

钻石票据攻击的核心在于利用域控制器（DC）颁发的真实票据授予票据（TGT），对其进行****解密、修改和重新加密，从而生成** 一个具有更高权限的 TGT。 黄金票据攻击和钻石票据攻击都需要访问 krbtgt 密钥。然而，钻石票据攻击需要访问 AES256 密钥。黄金票证攻击是利用伪造 票证授予票证（TGT），而钻石票证攻击则利用了域控制器（DC）请求的真实 TGT 进行解密和重新加密进行票据攻击。 黄金票据充分利用了能够从头开始伪造票据授予票据（TGT）的优势，而钻石票据则利用了能够解密和重新加密从域控制器（DC） **请求的真实 TGT 的优势。

钻石 PAC

PAC 是一种传递域控制器（DC）提供的授权相关信息的结构。身份验证协议使用 PAC 来验证身份以传输授权信息，从而控制对 资源的访问。说白了就是来验证身份是否有效的。最初的钻石 PAC 攻击远没有钻石票据灵活，攻击成功需要两个条件： 1、在没有特权属性证书（PAC）的情况下请求 TGT 2、确保要访问的服务的服务帐户未在其 UserAccountControl(UAC）属性中设置 NA 位

在服务帐户的 UAC 属性中设置 NA 位后，向该服务帐户请求的 ST 没有 PAC，攻击者下一个阶段就可以构造恶意 PAC 注入到 ST 中实现攻击。

由于 2021 年发布的补丁无法再在完全最新的域控制器（DC）上使用没有 PAC 的 TGT。我们知道需要拥有 KRBTGT 密钥，并且攻 击利用任何其他有效帐户来请求 TGT。所以我们想为什么不直接解密该 TGT，按照我们想要的方式修改它，重新计算 PAC 签 名，然后重新加密它。这就是我们所说的钻石票据。

钻石票据攻击利用

我们使用钻石票据的思路就是使用低权限的用户向域控请求一个普通的 TGT，随后进行解密修改签名和重新加密即可，当然这 些事情 Rubeus（基于 C# 专注于 Kerberos 票据操作的轻量化 mimikatz 子集）会自动帮我们完成，我们只需要提供信息即可。

信息收集

钻石票据相关面试题

什么是钻石票据攻击？

钻石票据攻击是一种基于 Kerberos 认证机制的高级攻击手段。它通过请求一个合法的票据授予票据（TGT），然后解密、修改和 重新加密该 TGT，生成一个具有更高权限的票据。与黄金票据（完全伪造 TGT）不同，钻石票据是基于真实 TGT 进行修改的，因 此更难以被检测。

钻石票据攻击与黄金票据攻击有什么区别？

攻击方式：黄金票据是完全伪造的 TGT，而钻石票据是基于真实 TGT 进行修改。 隐蔽性：钻石票据由于基于真实 TGT，因此在日志中看起来与正常 TGT 非常相似，更难以被检测。 所需条件：钻石票据需要 krbtgt 的 AES256 密钥，而黄金票据只需要 krbtgt 的 NTLM 哈希。

如何检测钻石票据攻击？

由于钻石票据基于真实 TGT 进行修改，检测难度较大。但可以通过以下方法进行检测： 监控异常权限变更：检查 TGT 中的 PAC 字段是否有异常的权限变更。 日志分析：分析域控制器的日志，查找异常的 TGT 请求和使用行为。

如何防御钻石票据攻击？

限制 krbtgt 密钥的访问：确保只有授权的管理员可以访问 krbtgt 的密钥。 启用 PAC 验证：在服务端启用 PAC 验证，增加伪造票据的难度。 监控和审计：对域控制器的日志进行实时监控和审计，及时发现异常行为。

五，蓝宝石票据

概述

蓝宝石票据与钻石票据类似，但票据不是伪造的，而是通过请求后获得的合法票据。蓝宝石票据的核心在于利用 S4U2Self + U2U 技术。攻击者首先请求一个普通用户的票据授予票据（TGT），然后通过 S4U2Self + U2U 机制获取高权限用户的合法 PAC（Privilege Attribute Certificate，特权属性证书）。接着，攻击者将高权限用户的 PAC 替换到普通用户的 TGT 中，从而生成一张具有高权限的合法票据。

S4U2self + U2U

为了在 Kerberos 协议层面对约束性委派进行支持，微软扩展了 S4U2self 协议，而使用 U2U 可以在没有 SPN 的情况下请求 S4U2Self。

    • Rubeus：轻量化，Kerberos 票据专用，免杀性强，内存执行友好；
    • impacket：跨平台，Kali 端生成蓝宝石票据 / 哈希传递，是 Windows 工具的补充。
  2. 核心避坑
    ：
    • 权限不足不操作，参数精准不遗漏（如域控全名、服务名称）；
    • 票据操作前清空旧票据，路径无中文 / 空格。
  3. 实战优先级
    ：
    • 隐蔽性：Rubeus（内存执行）> impacket（Kali）> 原版 mimikatz；
    • 功能全面性：mimikatz > impacket > Rubeus。
