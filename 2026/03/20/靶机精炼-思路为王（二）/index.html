[!IMPORTANT]

这段记录是****从拿到低权 shell 到“偷渡”成 root 的完整逃逸过程，每一步都有典型意义，下面按时间线拆开说。

  1. bash-3.2$ sudo ** **直接敲 sudo 没带任何参数，GNU sudo 会提示 -h 需要参数，等于没跑起来。
  2. bash-3.2$ sud ** **手抖少打字母，bash 报 command not found，纯粹 typo。
  3. bash-3.2$ su - ** 想切 root，但当前终端不是 TTY**（可能是通过 WebShell、反向 shell netcat 等进来的），** ** **su 拒绝交互输入 → **standard in must be a tty。
  4. bash-3.2$ python -c "import pty; pty.spawn('/bin/sh')" ** 经典升级交互式 TTY** 技巧：
    • **用 Python 的 **pty 模块重新分配一个伪终端；
    • **成功后提示符变成 sh-3.2$，现在有了“正常”终端。 ** 这一步是 90% 低权 shell 必做的“tty 逃逸”。
  5. sh-3.2$ su ** 再次执行 su，此时系统已认为它是终端设备**，可以交互输入密码。
  6. **输入密码 **50$cent ** 攻击者事先通过源码、配置文件、数据库或社工已拿到 root 口令；验证通过 → ** 提示符瞬间变成 ** [root@localhost ~]# ** 完成垂直提权。

一句话总结 ** ****“没 TTY → Python 造 PTY → 再 su → 口令正确 → root 到手”**，整套流程在实战里几十秒就能完成，是红队最常用、最朴实的提权场景之一。

渗透测试里“拿着公钥还能打”的 6 条现实路径

**思路一句话：**别跟数学过不去，去找“人”和“实现”的漏洞。

  1. 公钥 ➜ 用户名枚举 ➜ 社工/钓鱼
    • 把公钥扔进 GitHub、GitLab、HIBP、ssh-rank 等公开库搜一遍，经常能直接定位到开发者的个人账号、邮箱。
    • 拿到邮箱后走常规社工：克隆仓库投毒、发 PR 投毒、邮件钓鱼让开发者点恶意链接，最终拿到****他的私钥文件。
  2. 公钥指纹 ➜ 横向主机碰撞
    • **用 **ssh-keygen -lf key.pub 得到 16 字节 MD5 或 SHA256 指纹。
    • **在内网扫描所有 22 端口，把每台主机的 **ssh-rsa 主机密钥指纹也扫出来。
    • **如果开发机与生产机复用了同一对主机密钥，你就能确定“这台就是开发机”，继续打开发机上的个人 **.ssh/id_*。
  3. 弱随机数 / 弱素数生成器（ROCA、Debian-OpenSSL 事件）
    • 2017 年 Infineon TPM 的 RSA 密钥（ROCA）可以在小时级分解；
    • 2008 年 Debian OpenSSL 熵池补丁导致私钥空间只剩 2^15，能直接暴破。
    • **测试方法：用 **roca-detect 或 ssh-keyscan 抓公钥跑一遍，命中就能秒算私钥。
  4. 侧信道 / 故障注入（本地物理场景）
    • 如果你能物理接触目标笔记本，可以用电压毛刺、电磁注入让签名算法出错，
    • 拿到 1～2 条错误签名后就能用 lattice attack 还原私钥（参考 2021 TPM-Fail 论文）。
    • 属于高级硬件攻击，红队一般只在外包笔记本丢失场景用。
  5. 代理转发滥用（agent hijacking）
    • **开发者常开 **ssh -A 跳板机，把 ssh-agent 带进去。
    • **拿到普通用户权限后，直接读 **$SSH_AUTH_SOCK 套接字，就能调用 agent 做签名，无需私钥文件。
    • 工具：ssh-agent-hijack、gpg-agent-enum。
  6. 私钥文件残留（备份、镜像、CI）
    • **公钥出现在代码仓库 → 搜同一目录/历史提交，经常能拖出 **id_rsa 旧版本。
    • **搜 Docker 层、VMDK 快照、ELK 日志、Jenkins **credentials.xml，私钥文件明文躺尸概率极高。

linux 的 suid

**在 Linux 里，**SUID（Set User ID）是一种特殊的文件权限位。当普通用户执行带 SUID 位的可执行文件时，进程的有效 UID（euid）会被临时提升为该文件属主（owner）的 UID，从而拥有文件属主的权限。最典型的例子就是 /usr/bin/passwd：它属主是 root，且带 SUID，因此普通用户也能修改自己的 /etc/shadow 条目。

SUID 是 Linux 留给普通用户“临时借 root 身份”的后门， 用得好是便利工具，用不好就是本地提权的快车道。 **防御核心：**能去就去，不能去就换成 capabilities，再不行就 nosuid + 审计。

  • 远程：root 跑的 CGI / SSH / DHCP → 一条 HTTP 头或 DHCP 选项 = root。
  • 本地：SUID Bash / root cron → 设一个恶意环境变量 = root。** **修完 Bash 补丁 + 把 root 级脚本全换 dash，这条捷径就彻底堵死。
