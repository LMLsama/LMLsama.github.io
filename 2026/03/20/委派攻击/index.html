一、委派攻击概述

委派概述

域委派是指将域内用户的权限委派给服务(普通用户也能配置成服务账户和 WIN7$ 这种机器账户)账户，使得服务账户能以用户的权限在域内展开活动。简而言之，就是若用户 A 将自己的权限委派给服务账户 B，那么服务账户 B 就能以 A 的权限去访问其他服务。包括先斩后奏。

注：能够被委派的用户只能是服务账户或者机器账户。

服务账户 (Service Account)

服务账户（Service Account）是一种特殊的系统账户，用于运行特定的服务、应用程序或自动化任务，而不是用于人类用户的登录。服务器运行服务时所用的账户，用于运行服务并将其加入域内。常见的服务账户有 SQLServer、MYSQL、IIS 等。 服务账户通常用于运行特定的服务或应用程序，而这些服务或应用程序是运行在具体的机器（如服务器、工作站等）上的。因此，服务账户的运行****依赖于机器账户所代表的计算机环境。

机器账户 (Machine Account)

机器账户（Machine Account）是计算机在域环境中用于身份验证和管理的特殊账户。它代表了计算机本身的身份，与服务账户（用于运行服务或应用程序）不同，机器账户主要用于计算机与域控制器之间的交互。 **机器账号、计算机账号，所有加入域的主机都会有一个机器用户，用户名为机器名加 $，如：WIN7$、WINXP$。在域环境中，普通域用户最多可以创建 10 个计算机账户，但是本地账号无法创建计算机账户，机器用户就是一种域用户。 **默认情况下，加入域的机器默认在 CN=Computers 这个容器里面。

委派的类型

域委派分为三种：非约束性委派，约束性委派，基于资源的约束性委派。

总体来说委派是维权的一种手段。

二、非约束性委派攻击

概念

非约束委派（Unconstrained Delegation，UD）：服务账户可以获取被委派用户的 TGT，并将 TGT 缓存到 lsass 进程中，从而服务账户可使用该 TGT，模拟用户访问任意服务。

攻击者通过控制具有非约束性委派权限的主机，获取用户的 TGT，然后利用该 TGT 模拟用户身份访问其他服务。

流程

  1. 域内用户 A 经过 Kerberos 认证后访问 WEB 服务器上的资源或页面。
  2. WEB 服务器以服务账户 B 向 KDC 请求用户 A 的认证票据 TGT。
  3. KDC 检查服务账户 B 的委派属性，验证通过后下发 TGT。
  4. 服务账户 B 利用 TGT 向 KDC 的 TGS 申请服务票据（ST），用于访问文件服务器等功能。
  5. KDC 检查委派属性和申请的服务，下发对应服务的 ST。
  6. 服务账户 B 使用 ST 访问其他服务，以获取对应资源。
  7. 认证通过，获取到对应资源。
  8. Web 服务器返回资源给用户 A。

设置机器账户的委派

机器账户的非约束性委派设置机器账户 server 的属性中设置委派属性，勾选信任此计算机来委派任何服务..

攻击

Win7 木马上线并提权为 System 并且使用 mimikatz 导出票据

清空票据的核心目的是****消除旧票据干扰，保证 “权限测试结果真实”+“新导入票据唯一生效”；

这是 Kerberos 票据操作的通用最佳实践，不仅适用于非约束性委派，黄金 / 白银 / 蓝宝石票据攻击中也需先执行；

若不清空，极易出现 “票据导入成功但访问失败”“权限测试结果误判” 等问题，导致攻击流程卡壳。

privilege::debug 命令的本质是：调用 Windows API（AdjustTokenPrivileges），将 mimikatz 进程的 SeDebugPrivilege 权限从 “默认禁用状态” 切换为 “启用状态”；

防御手段

限制非约束性委派的使用

·尽量避免使用非约束性委派。如果确实需要委派权限，优先使用约束委派（ConstrainedDelegation）或资源约束委派 (Resource-BasedConstrainedDelegation）。约束委派可以限制服务账户只能访问特定的服务，从而降低攻击面。 ·例如，在域控制器上，不要随意勾选“信任此计算机进行委派”选项，而是明确指定允许委派的服务。

最小权限原则

·为服务账户分配最小必要权限，避免过度授权。即使攻击者获取了服务账户的权限，也无法轻易横向移动。

监控关键服务和账户

·对具有委派权限的账户和服务进行重点监控，例如监控登录事件（如事件 ID4768：TGT 请求）和委派事件（如事件 ID 4769：服务票据请求）。 ·使用安全信息和事件管理系统（SIEM）工具，如 Splunk、ELKStack 等，实时分析和告警异常行为。

定期审计委派配置

定期使用工具（如 Powerview、AdFind 等）扫描域内所有账户和服务的委派配置，检查是否存在不必要的非约束性委派。对发现的非约束性委派进行评估，如果不是必要，应立即禁用。

三、约束性委派

概念

约束委派（Constrained Delegation，CD）：由于非约束委派的不安全性，微软在 Windows Server 2003 中发布了约束性委派。

对于约束性委派，即 Kerberos 的两个扩展子协议 S4u2self (Service for User to Self) 和 S4u2Proxy (Service for User to Proxy)，服务账户只能获取用户的 ST（Service Ticket），从而只能模拟用户访问特定的服务。

约束性委派攻击的关键就是获得可转发的服务票据 ST，获取根据约束性委派的执行过程可知，只要控制配置约束性委派服务的机器，并获得了机器账户的密码或 Hash 等凭证，那么就可以劫持这台主机的 kerberos 请求过程，最终获得任意用户权限的 ticket。

配置了约束性委派的账户，其两个属性会被设置。

账户userAccountControl属性会被设置为TRUSTED_TO_AUTH_FOR_DELEGATION标志位，其值为16781312 账户的msDS-AllowedToDelegateTo属性，添加允许委派的服务

S4u2self 与 S4U2proxy 协议

S4u2self：允许服务代表任意用户请求访问自身服务的 ST 服务票据。后续票据先导出找到这个票据会很多很杂。 S4U2proXY：允许服务在已取得可转发的 ST 服务票据下，代表任意用户获取另一个服务的 ST 服务票据。约束委派限制了 S4U2proxy 协议的请求范围，使得配置了委派属性的服务只能模拟用户身份访问特定的其他服务。再去找 cifs 后续这个票据。

简单来说 S4u2self 协议使用用户的 TGT 向 KDC 请求用户的可转发的服务票据 ST1，而后 S4U2proxy 协议用这张 ST1 去发起 请求用于访问服务器的服务票据 ST2。

相关面试题** **

约束性委派攻击的基本原理是什么？

约束性委派攻击的核心在于攻击者利用服务账户的委派权限，以用户的身份访问目标服务。攻击者首先获取用户的 TGT（票据授 予票据），然后利用服务账户的委派权限请求目标服务的 ST（服务票据），从而以用户身份访问目标服务。

约束性委派攻击与非约束性委派攻击的主要区别是什么？

非约束性委派攻击：攻击者可以利用服务账户的权限访问域内任何服务，攻击面更广。 约束性委派攻击：攻击者只能访问被明确授权的服务，攻击面更窄，但仍然可以利用服务账户的权限进行横向移动。

如何检测约束性委派攻击？

1.监控事件日志：监控与委派相关的事件日志，如事件 ID4768（TGT 请求）和事件 ID4769（ST 请求）。异常的委派请求可能 表明攻击行为。 2.分析网络流量：使用网络监控工具（如 Wireshark）分析 Kerberos 流量，查找异常的票据请求。 3.定期审计委派配置：使用工具（如 PowerView、AdFind）扫描域内所有账户和服务的委派配置，检查是否存在未经授权 的委派。

如何识别异常的委派请求？

1.检查请求频率：频繁的委派请求可能表明攻击行为。 2.检查请求来源：检查委派请求的来源 IP 地址，是否来自已知的攻击者 IP。 3.检查目标服务：检查委派请求的目标服务是否为敏感服务（如域控制器、数据库服务）。

请描述一个约束性委派攻击的实际案例。

案例背景：某公司内部网络中，攻击者通过中间人攻击获取了用户的 TGT。 攻击过程： 1.攻击者找到一个具有约束性委派权限的服务账户。 2.攻击者利用该服务账户的权限，请求目标数据库服务的 ST。 3.攻击者使用获取的 ST 以用户身份访问数据库服务，获取敏感数据。 检测与响应： 1.安全团队通过监控事件日志发现异常的委派请求。 2.安全团队通过网络流量分析确认攻击路径。 3.安全团队隔离受影响的系统，修复漏洞，并加强监控和审计措施。

如何通过实际操作演示约束性委派攻击？

1.环境搭建：搭建一个包含域控制器、服务账户和目标服务的测试环境。 2.获取用户 TGT：通过中间人攻击工具（如 Responder）获取用户的 TGT。 3.利用服务账户委派权限：使用工具（如 Rubeus）请求目标服务的 ST。 4.访问目标服务：使用获取的 ST 以用户身份访问目标服务。 5.检测与防御：通过监控事件日志和网络流量，演示如何检测和防御攻击。
